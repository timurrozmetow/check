# Задание для Claude Code: «DirectorHub» — система управления задачами с панелью директора

## 1. Цель проекта

Построить веб-приложение для управления задачами в многокомпанейском бизнесе. Главная проблема: сейчас всё общение идёт через WhatsApp и превращается в кашу. Директор занят, много не читает — ему нужна максимально наглядная картина: какие задачи в работе, сколько % готово, что ждёт его решения. Админ (я) — центр системы: создаёт задачи, модерирует все обновления сотрудников, и только подтверждённая информация попадает на экран директора. В конце месяца система генерирует полный отчёт в Word.

В будущем будет мобильное приложение, поэтому API проектируем чисто и версионированно (`/api/v1`), без привязки логики к фронтенду.

Язык интерфейса: **только русский**. Все тексты UI, статусы, уведомления, отчёт — на русском.

---

## 2. Роли и права

### Админ (я)
- Создаёт проекты/компании и задачи, назначает исполнителей, ставит дедлайны и приоритеты.
- **Вручную выставляет % готовности задачи** — это единственный источник процента, сотрудники % не ставят.
- Модерирует все обновления от сотрудников: подтверждает (тогда обновление видно директору) или отклоняет с комментарием.
- Создаёт «запросы решения» для директора (например: «выберите офис из 3 вариантов»).
- Управляет пользователями (создание аккаунтов сотрудников, сброс пароля).
- Генерирует и скачивает месячный Word-отчёт.
- Видит всё.

### Директор
- Видит **только подтверждённую админом** информацию. Никаких черновиков и внутренней кухни.
- Главный экран — дашборд: KPI-карточки, прогресс-бары по задачам и проектам, блок «Ждут вашего решения».
- Принимает решения прямо на сайте: кнопки «Выбрать вариант», «Согласовать», «Отклонить» + необязательный комментарий.
- Может открыть задачу и посмотреть хронологию (только подтверждённые обновления) и прикреплённые файлы.
- Ничего не создаёт и не редактирует, кроме своих решений и комментариев.

### Сотрудник
- Видит только назначенные ему задачи.
- Отправляет обновления по задаче: текст + файлы (фото, PDF, Word, Excel). Каждое обновление уходит админу на модерацию со статусом «На проверке».
- Видит статус своих обновлений (на проверке / принято / отклонено с комментарием).
- Не видит чужие задачи, не видит % (или видит — реши сам, но по умолчанию видит), не видит панель директора.

## 3. Жизненный цикл задачи (workflow)

1. **Админ создаёт задачу**: название, описание, проект/компания, исполнитель(и), дедлайн, приоритет (низкий/средний/высокий/срочно). Автоматически фиксируется `created_at`.
2. **Статусы задачи**: `Новая` → `В работе` → `На проверке` → `Ждёт решения директора` (опционально) → `Завершена`. Также `Приостановлена` и `Отменена`. Статусы меняет только админ.
3. **Сотрудник работает** и отправляет обновления (текст + файлы). Обновление получает статус `pending`.
4. **Админ модерирует**: `approved` (появляется в хронологии задачи, видно директору) или `rejected` (сотрудник видит причину). При подтверждении админ может сразу обновить % готовности.
5. **Запрос решения директора**: админ создаёт `decision request` привязанный к задаче — заголовок, описание, список вариантов (каждый вариант: название, описание, файлы-вложения, например фото офисов) ИЛИ простое согласование (да/нет). Задача переходит в статус «Ждёт решения директора», у директора появляется уведомление и карточка в блоке «Ждут вашего решения».
6. **Директор решает**: выбирает вариант или жмёт «Согласовать»/«Отклонить», опционально пишет комментарий. Админ и исполнитель получают уведомление. Решение фиксируется в хронологии задачи.
7. **Завершение**: админ ставит статус `Завершена`, фиксируется `completed_at`. **Время выполнения = completed_at − created_at**, считается автоматически и попадает в отчёт.

Вся хронология задачи (создание, смена статусов, подтверждённые обновления, решения директора, изменения %) хранится как лента событий (activity log) — она же основа для отчёта.

## 4. Модули системы

### 4.1 Аутентификация
- Логин по email/логину + паролю. JWT (access + refresh), refresh в httpOnly cookie.
- Роли в токене. Middleware проверки роли на каждом защищённом роуте.
- Хеширование паролей — argon2.
- Регистрации нет: аккаунты создаёт админ. Seed-скрипт создаёт первого админа.

### 4.2 Проекты / Компании
- CRUD (только админ). Поля: название, цвет/иконка, описание, статус (активен/архив).
- Задачи всегда привязаны к проекту. У директора и админа — фильтр и группировка по проектам, сводный прогресс проекта (средний % его активных задач).

### 4.3 Задачи
- CRUD (админ). Поля: title, description, project_id, assignee_id (или несколько через связующую таблицу), status, progress (0–100, шаг 5), priority, deadline, created_at, completed_at.
- Карточка задачи: анимированный прогресс-бар, статус-бейдж, приоритет, дедлайн (с подсветкой просроченных), аватар исполнителя, хронология, файлы.

### 4.4 Обновления (модерация)
- Таблица `task_updates`: task_id, author_id, text, status (pending/approved/rejected), reject_reason, created_at, reviewed_at, reviewed_by.
- У админа — экран «Модерация» со счётчиком ожидающих, просмотром вложений, кнопками принять/отклонить и полем причины.

### 4.5 Решения директора
- Таблицы `decision_requests` (task_id, title, description, type: `choice` | `approval`, status: pending/decided, decided_at) и `decision_options` (request_id, title, description, is_selected).
- Файлы можно прикреплять и к запросу, и к каждому варианту.
- UI для директора: крупные карточки вариантов с фото-превью, одна большая кнопка выбора на каждой, поле комментария. Для согласований — две крупные кнопки «Согласовать» / «Отклонить». Минимум текста, максимум наглядности.

### 4.6 Файлы
- Загрузка через `@fastify/multipart`, хранение локально в `/uploads` (структура по годам/месяцам), раздача через Nginx.
- Разрешённые типы: изображения (jpg, png, webp), PDF, docx, xlsx. Лимит размера — 20 МБ, валидация MIME.
- Для изображений генерировать превью (sharp). В UI — лайтбокс для фото, иконки + скачивание для документов.

### 4.7 Уведомления (внутри сайта)
- Таблица `notifications`: user_id, type, title, body, link (роут внутри приложения), is_read, created_at.
- События: новая задача (сотруднику), новое обновление на модерацию (админу), обновление принято/отклонено (сотруднику), запрос решения (директору), решение принято (админу + исполнителю), приближается дедлайн.
- Колокольчик в шапке с бейджем непрочитанных, дропдаун, «отметить все прочитанными». Доставка: SSE (предпочтительно) или polling каждые 30 секунд — выбери SSE, если не усложняет.

### 4.8 Месячный Word-отчёт (максимально полный)
Генерация на бэкенде библиотекой **`docx`** (npm). Админ выбирает месяц (и опционально проект) → скачивает .docx. Содержимое:
1. **Титульная страница**: название компании/системы, «Отчёт о проделанной работе за <месяц год>», дата генерации.
2. **Сводка**: всего задач, завершено, в работе, просрочено, среднее время выполнения, средний % готовности активных задач.
3. **Графики** (рендерить как PNG через `chartjs-node-canvas` и вставлять картинками): задачи по статусам (donut), задачи по проектам (bar), завершено по неделям месяца (line).
4. **Таблица задач**: № / Задача / Проект / Исполнитель / Статус / % / Время выполнения (в днях и часах, человекочитаемо: «5 дн. 3 ч.») / Дедлайн (соблюдён/просрочен).
5. **Детализация по каждой завершённой задаче**: описание, хронология ключевых событий, принятые решения директора, **фото результатов** (подтверждённые изображения из обновлений, уменьшенные до разумного размера).
6. Стиль отчёта: строгий деловой, единый шрифт, колонтитулы с номерами страниц.

### 4.9 Дашборд директора (главная фича)
- KPI-карточки сверху: «Активных задач», «Завершено в этом месяце», «Ждут вашего решения» (акцентная, кликабельная), «Просрочено».
- Блок «Ждут вашего решения» — первым, с крупными карточками.
- Список задач: крупные карточки с анимированными прогресс-барами (заполнение анимируется при появлении), группировка/фильтр по проектам.
- Сводный прогресс каждого проекта — круговой индикатор.
- Всё считывается за 10 секунд без чтения длинных текстов.

## 5. Технический стек

**Frontend**: React 18 + Vite + TypeScript, shadcn/ui + Tailwind CSS, TanStack Query (серверное состояние), Zustand (UI-состояние), React Router, react-hook-form + zod, **framer-motion** (анимации), lucide-react, date-fns (локаль ru), sonner (тосты).

**Backend**: Node.js + **Fastify** + TypeScript, **Drizzle ORM** + MySQL 8, zod-валидация всех входных данных, @fastify/jwt, @fastify/multipart, @fastify/cookie, @fastify/rate-limit, sharp, docx, chartjs-node-canvas, pino (логи).

**Деплой**: PM2 + Nginx (reverse proxy, раздача /uploads и статики фронтенда). `.env` для конфигурации, `.env.example` в репозитории.

## 6. Архитектура и качество кода

Монорепозиторий:

```
directorhub/
├── server/
│   └── src/
│       ├── modules/            # модульная архитектура: каждый домен изолирован
│       │   ├── auth/           # routes.ts, service.ts, schemas.ts (zod)
│       │   ├── users/
│       │   ├── projects/
│       │   ├── tasks/
│       │   ├── updates/
│       │   ├── decisions/
│       │   ├── files/
│       │   ├── notifications/
│       │   └── reports/
│       ├── db/                 # schema.ts (Drizzle), migrations/, seed.ts
│       ├── plugins/            # jwt, cors, multipart, rate-limit, error-handler
│       ├── shared/             # типы, утилиты, константы статусов
│       └── app.ts / index.ts
├── client/
│   └── src/
│       ├── api/                # типизированный API-клиент + хуки TanStack Query
│       ├── components/         # ui/ (shadcn), common/
│       ├── features/           # tasks/, projects/, decisions/, moderation/, notifications/, reports/
│       ├── pages/              # по ролям: admin/, director/, employee/
│       ├── layouts/            # AdminLayout, DirectorLayout, EmployeeLayout
│       ├── stores/             # zustand: theme, ui
│       ├── lib/                # utils, constants
│       └── router.tsx          # защита роутов по ролям
└── package.json (workspaces)
```

Требования к коду:
- Строгий TypeScript (`strict: true`), никаких `any` без крайней необходимости.
- Слои на бэкенде: route (валидация + auth) → service (бизнес-логика) → Drizzle-запросы. Роуты тонкие.
- Единый формат ошибок API `{ error: { code, message } }`, глобальный error handler.
- Все статусы/роли — enum-константы в shared, никаких магических строк.
- REST API `/api/v1/...`, готовый к переиспользованию мобильным приложением.
- Транзакции там, где меняется несколько таблиц (например, решение директора: decision + task status + notifications + activity log).
- ESLint + Prettier.

## 7. Схема БД (Drizzle, MySQL)

Таблицы: `users` (id, name, email, password_hash, role: admin/director/employee, avatar, is_active), `projects`, `tasks`, `task_assignees` (task_id, user_id), `task_updates`, `decision_requests`, `decision_options`, `files` (полиморфная привязка: entity_type + entity_id, path, thumb_path, mime, size, original_name), `notifications`, `activity_log` (task_id, actor_id, type, payload JSON, created_at).

Индексы по внешним ключам, статусам и датам (запросы отчёта фильтруют по месяцу `completed_at`/`created_at`).

## 8. Дизайн: анимированный офисный сайт

- **Светлая + тёмная тема с переключателем** (next-themes-подход через class на html, тема сохраняется). Оба режима должны выглядеть законченно, не «инверсия ради галочки».
- Стиль: современный офисный дашборд — чистый, много воздуха, аккуратные карточки с мягкими тенями, скруглениями, один акцентный цвет.
- Анимации (framer-motion), со вкусом и быстро (150–300 мс), без перегруза:
  - плавное появление карточек списком (stagger);
  - прогресс-бары анимированно заполняются до значения при появлении;
  - числа в KPI-карточках «набегают» (count-up);
  - переключение темы — плавный переход;
  - модалки/дропдауны — scale + fade;
  - колокольчик слегка «звенит» при новом уведомлении.
- Полностью адаптивный: директор будет открывать с телефона — мобильная версия дашборда обязательна и приоритетна.
- Русская локализация дат (date-fns/locale/ru), формат «7 июля 2026».

## 9. Порядок реализации (по этапам, с рабочим результатом после каждого)

1. Каркас монорепо, Fastify + Drizzle + миграции + seed (админ, директор, 2 сотрудника, 2 проекта, 6 демо-задач), auth + роли, каркас клиента с layout'ами по ролям и переключателем темы.
2. Проекты + задачи: CRUD админа, списки у всех ролей, прогресс-бары, статусы.
3. Обновления сотрудников + загрузка файлов + экран модерации админа.
4. Запросы решений + экран решений директора + дашборд директора целиком.
5. Уведомления (SSE + колокольчик).
6. Word-отчёт с графиками и фото.
7. Полировка: анимации, адаптив, пустые состояния, скелетоны загрузки, обработка ошибок.

После каждого этапа проект должен запускаться и работать. Начни с этапа 1 и двигайся по порядку.

## 10. Критерии готовности

- Три роли работают со своими правами, чужие данные недоступны (проверка на уровне API, не только UI).
- Директор за 10 секунд видит: общий прогресс, что ждёт его решения, и может решить в 2 клика.
- Ни одно неподтверждённое обновление не видно директору.
- Время выполнения задач считается автоматически и корректно попадает в отчёт.
- Word-отчёт открывается в MS Word без ошибок, содержит титул, сводку, 3 графика, таблицу и детализацию с фото.
- Тёмная и светлая темы полноценны, сайт адаптивен, анимации плавные.
- Код модульный, типизированный, без дублирования; `.env.example` и README с инструкцией запуска (dev + PM2/Nginx prod).
